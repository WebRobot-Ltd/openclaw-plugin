# WebRobot Python Extensions

Python Extensions are the lightweight extensibility path for WebroBot pipelines. Instead of building a compiled Scala plugin, you write a Python function that the ETL engine wraps as a Spark UDF at runtime. The engine generates PySpark boilerplate around your function automatically.

Use Python Extensions when the logic is:
- Specific to one pipeline or agent
- Generated by an AI agent at pipeline-definition time
- Frequently changing (no build/deploy cycle)

Use a **Scala ETL plugin** (see `/webrobot-plugin-dev`) when the stage must be reused across organizations, versioned independently, or requires direct DB access via `WebroStageContext` (e.g. `ctx.query`, `ctx.execute`, `ctx.llm`). Python Extensions run as Spark UDFs without `WebroStageContext` — no DB, no LLM facade, no MinIO.

Before naming a Python Extension, check the catalog so the name doesn't collide with an existing native or plugin stage:

```bash
curl -fsSL "https://api.webrobot.eu/webrobot/api/catalog/stages?stage_name=my_transform"
```

The catalog endpoint is public (no auth required, read-only).

## Function contract

Every Python Extension function receives one row (a `dict`) and must return a dict containing **all original fields plus any new fields**. Never drop fields unless intentional.

```python
def my_transform(row):
    # row is a plain dict — all pipeline fields are present
    value = row.get('some_field', '')
    result = value.upper()
    return {
        **row,               # preserve all existing fields
        'new_field': result  # add or override fields
    }
```

Rules:
- No `spark` or `SparkSession` in the signature — the engine injects context automatically
- Use only stdlib + pre-installed packages (requests, bs4, re, json, math)
- Do not raise exceptions — return a default value and log warnings instead
- The function name in the YAML key must match the `def` name exactly

## Extension types

| Type | YAML key | Use case |
|------|----------|----------|
| `row_transform` | `python_row_transform:name` | Transform fields of one row → same row |
| `resolver` | `python_resolver:name` | Extract an attribute from a page element |
| `validator` | `python_validator:name` | Validate and clean a row |

`row_transform` is the most common type. Use it for parsing, normalization, scoring, enrichment.

## Three modes

### Mode A — Inline YAML

Extension code is embedded directly in the pipeline YAML under `python_extensions.stages`. Best for prototyping and single-use pipelines.

```yaml
pipeline:
  - stage: python_row_transform:parse_price_inline
    args: []
  - stage: python_row_transform:normalize_category_inline
    args: []

python_extensions:
  stages:
    parse_price_inline:
      type: row_transform
      function: |
        def parse_price_inline(row):
            import re
            price_str = str(row.get('price', '0')).strip()
            currency_map = {'€': 'EUR', '$': 'USD', '£': 'GBP'}
            currency = 'EUR'
            for sym, code in currency_map.items():
                if sym in price_str:
                    currency = code
                    price_str = price_str.replace(sym, '')
                    break
            price_str = price_str.replace(',', '.').strip()
            m = re.search(r'(\d+(?:\.\d+)?)', price_str)
            amount = float(m.group(1)) if m else 0.0
            return {**row, 'price_amount': amount, 'price_currency': currency}

    normalize_category_inline:
      type: row_transform
      function: |
        def normalize_category_inline(row):
            cat = str(row.get('category', 'Other'))
            parts = cat.split(' > ') if ' > ' in cat else [cat]
            parts = [p.strip().title() for p in parts if p.strip()]
            return {
                **row,
                'main_category': parts[0] if parts else 'Other',
                'sub_category': parts[1] if len(parts) > 1 else None
            }
```

### Mode B — Database (registered)

Extensions are registered once via API and referenced by name in the YAML. Best when multiple agents share the same logic.

Register:
```bash
curl -X POST "$WEBROBOT_API/webrobot/api/python-extensions/register" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "agentId": "123",
    "name": "parse_price",
    "code": "def parse_price(row):\n    ...\n    return {**row, ...}",
    "type": "STAGE",
    "status": "ACTIVE"
  }'
```

Reference in YAML:
```yaml
pipeline:
  - stage: python_row_transform:parse_price
    args: []
```

### Mode C — Hybrid (recommended for production)

Common logic comes from the DB; pipeline-specific logic is inline. Best balance of reusability and flexibility.

```yaml
pipeline:
  - stage: python_row_transform:parse_price        # from DB
    args: []
  - stage: python_row_transform:domain_specific    # inline below
    args: []

python_extensions:
  stages:
    domain_specific:
      type: row_transform
      function: |
        def domain_specific(row):
            # logic specific to this pipeline only
            return {**row, 'processed': True}
```

## Uploading a YAML with inline extensions

```bash
curl -X POST "$WEBROBOT_API/webrobot/api/python-extensions/process-yaml" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "agentId": "123",
    "yamlContent": "<full yaml content>"
  }'
```

The API parses the `python_extensions` block, generates PySpark wrapper code, and saves it to `agent.pyspark_code`. The generated code is then used when the job executes.

## Decision guide

```
Need custom logic in a pipeline?
    │
    ├─ Prototyping / one-off? ──────────────────────── Mode A (inline)
    │
    ├─ Reused by multiple agents / teams? ──────────── Mode B (DB)
    │
    ├─ Mix: shared base + pipeline-specific tweaks? ── Mode C (hybrid) ⭐ recommended
    │
    └─ Logic needs DB access / cross-org reuse?
       └── Build a Scala ETL plugin instead (/webrobot-plugin-dev)
```

## Common patterns

### Parse a price field
```python
def parse_price(row):
    import re
    s = str(row.get('raw_price', '')).strip()
    s = s.replace(',', '.').replace('€', '').replace('$', '').strip()
    m = re.search(r'(\d+(?:\.\d+)?)', s)
    return {**row, 'price': float(m.group(1)) if m else None}
```

### Compute a derived field
```python
def add_discount_pct(row):
    original = row.get('original_price') or 0
    current  = row.get('current_price') or 0
    pct = round(((original - current) / original) * 100, 2) if original > 0 else 0.0
    return {**row, 'discount_pct': pct}
```

### Fail-safe enrichment
```python
def enrich_availability(row):
    raw = str(row.get('availability_text', '')).lower()
    if 'stock' in raw or 'disponib' in raw:
        status = 'in_stock'
    elif 'esaurit' in raw or 'out' in raw:
        status = 'out_of_stock'
    else:
        status = 'unknown'
    return {**row, 'availability': status}
```

## Workflow when building a Python Extension

1. Understand what fields are already in the row — check the preceding stages in the pipeline YAML
2. Write the function locally and test it with a sample `row` dict (plain Python, no Spark needed)
3. Decide the mode: inline for quick work, DB if it will be reused
4. Embed the function under `python_extensions.stages` (Mode A/C) or register via API (Mode B)
5. Reference it in the pipeline stages as `python_row_transform:<function_name>`
6. Validate the full YAML with `validate_manifest` before deploying
